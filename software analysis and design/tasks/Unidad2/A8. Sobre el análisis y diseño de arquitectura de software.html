A8. Sobre el análisis y diseño de arquitectura de software

📋 Descripción

En esta actividad aplicarás criterios técnicos para seleccionar y justificar arquitecturas de software ante distintos contextos de negocio, diseñarás la estructura interna de un sistema usando la arquitectura hexagonal y documentarás las decisiones de diseño más relevantes mediante Architecture Decision Records (ADRs).

El caso de estudio es el Sistema de Biblioteca Digital, que permite a lectores buscar libros, realizar préstamos y hacer reservas; a bibliotecarios registrar devoluciones y gestionar el inventario; y al administrador configurar el sistema y generar reportes de actividad.

🎯 Objetivos de Aprendizaje
  1. Seleccionar y justificar una arquitectura de software apropiada según los atributos de calidad del sistema.
  2. Diseñar la estructura hexagonal de un sistema identificando puertos primarios, secundarios y sus adaptadores correspondientes.
  3. Documentar decisiones arquitectónicas mediante ADRs siguiendo el formato estándar (contexto, decisión, consecuencias, alternativas).
📝 Enunciado del caso

El Sistema de Biblioteca Digital debe ser diseñado considerando cuatro contextos distintos:

  • Escenario A — Startup MVP: Equipo de 3 desarrolladores, primer lanzamiento, ~1 000 usuarios esperados, presupuesto reducido.
  • Escenario B — Sistema bancario crítico: Alta disponibilidad 99.9 %, equipos independientes por módulo, regulaciones de auditoría.
  • Escenario C — Sistema interno de RRHH: Empresa mediana, cambios de negocio frecuentes, integraciones con sistemas legados (ERP, nómina).
  • Escenario D — E-commerce con picos estacionales: Carga variable (x10 en temporadas altas), bajo costo en reposo, escalado automático requerido.

Para el diseño hexagonal y los ADRs utiliza el contexto completo del sistema (todos los actores y flujos).

💡 Solución de Referencia
Ejercicio 1 — Análisis de Arquitectura

Escenario A — Startup MVP (3 devs, 1 000 usuarios)

✅ Arquitectura seleccionada: Monolítica en Capas (N-Tier)

Factor Análisis
Tamaño del equipo 3 desarrolladores no pueden mantener una infraestructura compleja
Fase del producto MVP requiere velocidad de desarrollo por encima de escalabilidad
Carga esperada 1 000 usuarios es manejable con un monolito bien optimizado
Presupuesto Menor infraestructura = menor costo operacional

Monolito modular con capas: Presentation → Application → Domain → Infrastructure. Framework: Next.js o NestJS + React. 

app/
├── api/            # Capa de presentación (rutas HTTP)
├── services/       # Capa de aplicación (casos de uso)
├── domain/         # Capa de dominio (entidades y reglas)
└── infrastructure/ # Capa de infraestructura (DB, emails)

Escenario B — Sistema bancario crítico (50 devs, millones de transacciones)

✅ Arquitectura seleccionada: Microservicios + Clean Architecture por servicio

Factor Análisis
Tamaño del equipo 50 desarrolladores necesitan autonomía por dominios (equipos de 5–8)
Criticidad Sistema bancario requiere resiliencia, aislamiento de fallos
Escala Millones de transacciones requieren escalado independiente por servicio
Cumplimiento Servicios aislados facilitan auditorías (PCI-DSS) 
API Gateway
├── accounts-service      # Cuentas bancarias
├── transactions-service  # Transacciones (escalado horizontal)
├── auth-service          # Autenticación y autorización
├── notifications-service # Emails, SMS, push
├── reports-service       # Generación de reportes
└── audit-service         # Registro de auditoría (crítico)

Comunicación asíncrona con Apache Kafka. Cada servicio implementa Clean Architecture internamente.

Escenario C — Sistema interno de RRHH (200 empleados, bajo presupuesto)

✅ Arquitectura seleccionada: Monolítica Modular

Factor Análisis
Usuarios 200 empleados internos = carga muy baja y predecible
Presupuesto Bajo presupuesto excluye infraestructura de microservicios
Complejidad RRHH no justifica la complejidad de servicios distribuidos
Disponibilidad Sistema interno tolera mantenimientos programados 
hrms-app/
├── modules/
│   ├── employees/   # Gestión de empleados
│   ├── payroll/     # Nómina
│   ├── leave/       # Gestión de permisos y vacaciones
│   └── reports/     # Reportes de RRHH
├── shared/          # Utilidades comunes
└── config/

Desplegado en PaaS económico: Railway, Render o Heroku.

Escenario D — E-commerce con picos estacionales (tráfico 10–50×)

✅ Arquitectura seleccionada: Hexagonal + Escalado Horizontal Selectivo

Factor Análisis
Picos de tráfico Black Friday, Navidad → tráfico 10–50× el normal
Operaciones críticas Checkout y pagos deben ser siempre disponibles
Consistencia Inventario y pagos requieren consistencia fuerte
Costo No justifica microservicios completos; sí justifica escalado del área crítica

Arquitectura híbrida: monolito hexagonal + servicios independientes escalables (checkout-service, payments-service). Complementado con CDN + Redis + Auto-scaling + Circuit breaker.

Ejercicio 3 — Architecture Decision Records (ADRs)
ADR-001 — Base de datos: PostgreSQL vs MongoDB
Estado Aceptado
Contexto El sistema maneja datos relacionales (Libro↔Préstamo↔Usuario↔Multa) con JOINs complejos para reportes. Los datos son estructurados y las relaciones son parte del dominio.
Decisión PostgreSQL con Prisma ORM
Consecuencias ✅ ACID nativo, foreign keys, ecosistema maduro con Prisma, soporte JSON para datos flexibles
Consecuencias ⚠️ Schema rígido requiere migraciones, escalado horizontal complejo
Alternativas MongoDB (schema flexible pero datos duplicados); MySQL (similar pero menos soporte JSON)
ADR-002 — Autenticación: JWT vs Sesiones
Estado Aceptado
Contexto 3 roles (Lector, Bibliotecario, Admin), API REST + posible app móvil futura, sesiones con expiración configurable.
Decisión JWT con Access Token (15 min) + Refresh Token (7 días lectores / 8h admin). Refresh token en HttpOnly cookie, almacenado en BD para revocación.
Consecuencias ✅ Stateless, multi-plataforma, rol incluido en token, revocación controlada
Consecuencias ⚠️ Complejidad de dos tokens, access token no revocable instantáneamente, payload mayor
Alternativas Sesiones + Redis (revocación instantánea pero stateful, overhead Redis); OAuth2 (incompatible con carné institucional)
ADR-003 — Almacenamiento de archivos: Local vs AWS S3
Estado Aceptado
Contexto Reportes PDF generados + posibles portadas de libros. Volumen bajo en MVP.
Decisión Almacenamiento local para MVP con interfaz IFileStorageService para migración futura
Consecuencias ✅ Costo cero, simple, protegido por DIP, volumen adecuado para MVP
Consecuencias ⚠️ No escalable horizontalmente, sin CDN, riesgo de pérdida si no hay backup
Alternativas AWS S3 ($0.023/GB, escala producción); Cloudinary (solo imágenes, no PDFs)
Plan de migración Implementar S3FileStorageService → intercambiar en IoC → sin cambios en dominio
ADR-004 — Framework frontend: Next.js vs Remix
Estado Aceptado
Contexto Catálogo público (SEO requerido), panel usuario autenticado y panel administrador.
Decisión Next.js 14+ con App Router
Consecuencias ✅ SSR/SSG nativo, API Routes integradas, ecosistema masivo, despliegue en Vercel, React Server Components
Consecuencias ⚠️ Complejidad del App Router, opinionado hacia Vercel, bundle mayor para apps interactivas
Alternativas Remix (mejor manejo de formularios, comunidad menor); SPA React+Vite (sin SSR, mal SEO)
🎓 Conceptos Clave
Concepto Descripción breve
Atributos de calidad Características no funcionales (escalabilidad, mantenibilidad, disponibilidad) que guían la selección de una arquitectura
Arquitectura monolítica en capas Estructura N-Tier (Presentación → Aplicación → Dominio → Infraestructura) en un único despliegue
Arquitectura de microservicios Sistema compuesto de servicios independientes, desplegables y escalables por separado
Arquitectura hexagonal (Ports & Adapters) Patrón que aísla el núcleo de dominio de los detalles externos mediante puertos (interfaces) y adaptadores (implementaciones)
Puerto primario Interfaz de entrada al dominio; define lo que el sistema ofrece (ej. casos de uso)
Puerto secundario Interfaz de salida del dominio; define lo que el sistema necesita (ej. repositorios, notificaciones)
Adaptador Implementación concreta de un puerto; conecta el dominio con tecnologías externas (HTTP, BD, email)
Principio de Inversión de Dependencias (DIP) Las capas de alto nivel no dependen de capas de bajo nivel; ambas dependen de abstracciones (interfaces)
Separación de preocupaciones (SoC) Cada capa o componente tiene una única responsabilidad bien definida
Architecture Decision Record (ADR) Documento breve que registra una decisión técnica importante: contexto, opciones evaluadas y consecuencias
Trade-off analysis Evaluación explícita de las ventajas y desventajas de cada alternativa antes de tomar una decisión de diseño
