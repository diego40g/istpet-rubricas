A2. Identificación de Requisitos para un Sistema de Reservas de Restaurant

📋 Descripción

A partir del enunciado narrativo del Sistema de Reservas de Restaurant, identifica, clasifica y prioriza los requisitos funcionales (RF) y no funcionales (RNF) del sistema. Documenta cada requisito en formato tabular estándar e incluye notas de análisis que justifiquen tus decisiones de priorización.

🎯 Objetivos de Aprendizaje
  1. Diferenciar entre requisitos funcionales y no funcionales aplicados a un sistema real.
  2. Clasificar requisitos según prioridad (Alta / Media / Baja) usando criterios objetivos.
  3. Definir el conjunto mínimo viable (MVP) del sistema.
  4. Escribir requisitos verificables con métricas concretas.
📝 Requerimientos

Produce los siguientes entregables a partir del caso de estudio:

Entregable Mínimo requerido
Tabla de Requisitos Funcionales 8 RF con ID, descripción y prioridad
Tabla de Requisitos No Funcionales 5 RNF con ID, descripción, categoría y prioridad
Notas de Análisis 3 observaciones que justifiquen decisiones de priorización
Enunciado del caso
El restaurante "La Mesa" necesita un sistema web que permita a sus clientes hacer reservas en línea. Los clientes deben poder consultar la disponibilidad de mesas por fecha y hora, crear reservas indicando el número de personas, y cancelarlas si cambian de planes. El sistema debe enviarles un correo de confirmación y un recordatorio 24 horas antes. El personal del restaurante necesita ver un panel con todas las reservas del día. El administrador debe poder configurar las mesas disponibles y generar reportes de ocupación.
Entrega: Sube el documento con las tablas de requisitos y las notas de análisis antes de la fecha límite.

💡 Solución de Referencia
Requisitos Funcionales
ID Descripción Prioridad
RF-001 El sistema debe permitir a los clientes consultar la disponibilidad de mesas por fecha y hora. Alta
RF-002 El sistema debe permitir a los clientes crear una reserva indicando fecha, hora y número de personas. Alta
RF-003 El sistema debe enviar un email de confirmación al cliente tras realizar una reserva exitosa. Alta
RF-004 El sistema debe permitir a los clientes cancelar una reserva existente. Alta
RF-005 El sistema debe requerir registro e inicio de sesión para realizar o cancelar reservas. Alta
RF-006 El sistema debe permitir al personal del restaurante ver todas las reservas del día en un panel. Alta
RF-007 El sistema debe enviar un recordatorio por email al cliente 24 horas antes de su reserva. Media
RF-008 El sistema debe permitir al administrador configurar el número máximo de mesas disponibles y su capacidad. Media
RF-009 El sistema debe mostrar un historial de reservas anteriores al cliente autenticado. Baja
RF-010 El sistema debe generar reportes de ocupación semanal y mensual para el administrador. Baja
Requisitos No Funcionales
ID Descripción Categoría Prioridad
RNF-001 El sistema debe cargar cualquier página en menos de 2 segundos bajo condiciones normales de uso. Rendimiento Alta
RNF-002 El sistema debe estar disponible el 99.5 % del tiempo (máx. 3.6 h de inactividad/mes). Disponibilidad Alta
RNF-003 Las contraseñas de los usuarios deben almacenarse usando hash bcrypt con salt único por usuario. Seguridad Alta
RNF-004 La interfaz debe ser accesible desde dispositivos móviles (diseño responsivo). Usabilidad Alta
RNF-005 El sistema debe soportar al menos 200 usuarios concurrentes sin degradación de rendimiento. Escalabilidad Media
RNF-006 El sistema debe cumplir con la Ley de Protección de Datos Personales vigente en el país. Seguridad Alta
RNF-007 El código fuente debe seguir estándares de documentación para facilitar el mantenimiento futuro. Mantenibilidad Media
Notas de Análisis
  • Los requisitos RF-001 al RF-006 representan el MVP del sistema: sin ellos el negocio no puede operar en línea. Deben ser implementados en la primera entrega.
  • RNF-001 y RNF-002 son críticos para la experiencia del usuario: un sistema lento o caído durante el servicio del restaurante (horario pico: 12h–15h y 19h–23h) genera pérdida directa de reservas.
  • RNF-006 debe revisarse con asesoría legal antes del lanzamiento, ya que el incumplimiento implica sanciones económicas para el negocio.
🎓 Conceptos Clave
Concepto Descripción breve
Requisito Funcional (RF) Describe qué hace el sistema: servicios, acciones, transformaciones. Pregunta guía: ¿Qué hace el sistema cuando el usuario realiza esta acción?
Requisito No Funcional (RNF) Describe cómo se comporta el sistema: atributos de calidad medibles (rendimiento, seguridad, usabilidad).
Priorización MoSCoW Alta = Must Have (MVP) · Media = Should Have (v2) · Baja = Could Have (deseable)
Verificabilidad Un RNF es verificable si se puede diseñar una prueba objetiva para medirlo (ej.: prueba de carga, uptime).
