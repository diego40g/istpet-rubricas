A14. DevOps básico, CI/CD y automatización de calidad

📋 Descripción

En esta actividad configurarás el flujo de entrega continua del Sistema de Biblioteca Digital usando herramientas de DevOps reales. Automatizarás las verificaciones de calidad tanto en el entorno local (hooks de pre-commit y pre-push) como en la nube (pipeline de GitHub Actions), y documentarás el estado del proyecto con badges visibles en el README.

Al finalizar, el repositorio contará con un ciclo profesional: cada commit es verificado localmente por ESLint y Prettier, cada push ejecuta los tests, y cada integración a main dispara un pipeline que valida lint, cobertura y compilación antes de permitir el merge.

🎯 Objetivos de Aprendizaje
  1. Configurar un pipeline de CI/CD con GitHub Actions que ejecute lint, tests y build de forma automática en cada push o pull request.
  2. Implementar hooks de pre-commit y pre-push con Husky para garantizar calidad del código antes de cada integración.
  3. Documentar el estado del repositorio con badges de CI y cobertura visibles en el README.
📝 Enunciado del caso

El repositorio del Sistema de Biblioteca Digital (creado en la actividad A13) debe tener su flujo de calidad completamente automatizado:

  • Pipeline CI: Cada push a main o develop, y cada pull request a main, debe disparar el pipeline con los 3 jobs en secuencia. Si alguno falla, los siguientes no se ejecutan y no se puede hacer merge.
  • Hooks locales: Antes de cada git commit, lint-staged debe formatear y verificar solo los archivos modificados. Antes de cada git push, deben ejecutarse la verificación de tipos y la suite completa de tests.
  • README: El archivo README.md debe mostrar el badge de estado del CI (verde/rojo) y el porcentaje de cobertura actualizado automáticamente por Codecov.

Ejercicio 1 — Configurar GitHub Actions

Archivo .github/workflows/ci.yml 

name: CI Pipeline — Biblioteca Digital

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  NODE_VERSION: "20"

jobs:
  # ── JOB 1: Lint y análisis estático ─────────────────────────────────────
  lint:
    name: 🔍 Lint & Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "${{ env.NODE_VERSION }}", cache: "npm" }
      - run: npm ci
      - run: npm run format:check
      - run: npm run lint
      - run: npm run type-check

  # ── JOB 2: Tests + cobertura ────────────────────────────────────────────
  test:
    name: 🧪 Unit Tests & Coverage
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "${{ env.NODE_VERSION }}", cache: "npm" }
      - run: npm ci
      - run: npm run test:coverage
      - uses: codecov/codecov-action@v3
        with:
          files: ./coverage/lcov.info
          flags: unittests
          fail_ci_if_error: false
      - uses: actions/upload-artifact@v4
        with: { name: coverage-report, path: coverage/, retention-days: 7 }

  # ── JOB 3: Build ────────────────────────────────────────────────────────
  build:
    name: 🏗️ Build
    runs-on: ubuntu-latest
    needs: [lint, test]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "${{ env.NODE_VERSION }}", cache: "npm" }
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with: { name: dist, path: dist/, retention-days: 1 }

Flujo de ejecución de jobs 

push/PR a main o develop
         │
         ▼
┌─────────────────┐
│  🔍 lint        │  ← ESLint + Prettier + tsc --noEmit
└────────┬────────┘
         │ si pasa
         ▼
┌─────────────────┐
│  🧪 test        │  ← Jest --coverage + Codecov
└────────┬────────┘
         │ si pasa
         ▼
┌─────────────────┐
│  🏗️ build       │  ← tsc + artefacto dist/
└─────────────────┘
         ✅ Pipeline completado

Protección de rama (Recomendado)

En GitHub → Settings → Branches → Add rule para main:

  • ✅ Require status checks to pass: seleccionar lint, test, build
  • ✅ Require branches to be up to date
  • ✅ Require a pull request before merging

Ejercicio 2 — Pre-commit Hooks con Husky

Instalación y configuración 

# 1. Instalar
npm install -D husky lint-staged
npx husky install

# 2. Crear hooks
npx husky add .husky/pre-commit "npx lint-staged"
npx husky add .husky/pre-push "npm run type-check && npm test"

# 3. Dar permisos
chmod +x .husky/pre-commit .husky/pre-push

# 4. Commitear configuración
git add .husky/ package.json
git commit -m "chore: configure husky hooks"

Agregar a package.json 

{
  "scripts": { "prepare": "husky install" },
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md,yml,yaml}": ["prettier --write"]
  }
}

Archivo .husky/pre-commit 

#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"
echo "🔍 Ejecutando lint-staged..."
npx lint-staged
echo "✅ Pre-commit checks passed!"

Archivo .husky/pre-push 

#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"
echo "🔎 Verificando tipos TypeScript..."
npm run type-check
echo "🧪 Ejecutando tests..."
npm test
echo "✅ Pre-push checks passed!"

Demostración: commit que falla vs. commit que pasa 

# CASO 1 — Código con error (debe fallar)
cat > src/test-error.ts << 'EOF'
const unusedVariable = "no se usa";        // error: unused var
function bad(a: any) { return a == null; } // error: == y any
EOF
git add src/test-error.ts
git commit -m "test: debe fallar"
# → ✖ eslint --fix [FAILED] — el commit NO se realiza

# CASO 2 — Código correcto (debe pasar)
rm src/test-error.ts
cat > src/greet.ts << 'EOF'
export const greet = (name: string): string => `Hola, ${name}!`;
EOF
git add src/greet.ts
git commit -m "feat: add greet"
# → ✅ Pre-commit checks passed! — commit creado
Hook Cuándo Verifica Si falla
pre-commit git commit ESLint + Prettier (solo archivos staged) Commit NO se crea
pre-push git push TypeScript tipos + suite de tests Push NO se realiza

Para saltarse en emergencias: git commit --no-verify / git push --no-verify. ⚠️ El pipeline de CI atrapará los errores igualmente.

🎓 Conceptos Clave
Concepto Descripción breve
DevOps Cultura y conjunto de prácticas que unifican el desarrollo (Dev) y las operaciones (Ops) para entregar software de manera continua
CI (Integración Continua) Práctica de integrar cambios frecuentemente al repositorio principal, con verificación automática en cada integración
CD (Entrega Continua) Práctica de mantener el código siempre en un estado desplegable; cada cambio que pasa CI puede ir a producción
GitHub Actions Plataforma de automatización integrada en GitHub; ejecuta workflows definidos en YAML ante eventos del repositorio
Workflow (flujo de trabajo) Archivo YAML en .github/workflows/ que define los triggers, jobs y steps del pipeline de CI/CD
Job Unidad de trabajo en GitHub Actions que se ejecuta en una máquina virtual independiente (runs-on: ubuntu-latest)
Step Paso dentro de un job: puede ser un comando shell (run:) o una action reutilizable (uses:)
needs Dependencia entre jobs; un job con needs: lint solo se ejecuta si el job lint terminó exitosamente
Artefacto de CI Archivo generado durante el pipeline (reporte de cobertura, binario compilado) guardado con actions/upload-artifact
Hook de Git Script que se ejecuta automáticamente en eventos del flujo Git (pre-commit, pre-push, commit-msg)
Husky Librería que facilita la configuración de hooks de Git en proyectos Node.js
lint-staged Herramienta que ejecuta linters solo sobre los archivos que están en el área de stage de Git, haciendo el pre-commit muy rápido
Protección de rama Regla en GitHub que impide el merge a main si los checks de CI no pasan o si no hay revisión de código aprobada
Badge de CI Imagen dinámica en el README que muestra el estado actual del pipeline (passing/failing) actualizada en cada ejecución
Codecov Plataforma que recibe reportes de cobertura de tests y muestra un badge con el porcentaje actualizado
Conventional Commits Convención de mensajes de commit estructurados (feat:, fix:, ci:, chore:) que facilitan la generación automática de changelog
