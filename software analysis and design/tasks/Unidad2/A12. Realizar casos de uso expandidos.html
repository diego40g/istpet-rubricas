A12. Realizar casos de uso expandidos

📋 Descripción

En esta actividad especificarás los casos de uso expandidos del Sistema de Biblioteca Digital siguiendo el estándar IEEE 29148:2018. Partirás de un diagrama general de casos de uso que identifique a todos los actores y las principales funcionalidades, y luego desarrollarás tres especificaciones detalladas con flujo principal, flujos alternativos numerados, precondiciones, postcondiciones y las relaciones <<include>> y <<extend>> justificadas.

🎯 Objetivo de Aprendizaje
  1. Redactar casos de uso expandidos con flujos principales, flujos alternativos y condiciones de extensión/inclusión según IEEE 29148:2018.
📝 Enunciado del caso

El Sistema de Biblioteca Digital tiene los siguientes actores:

  • Lector (actor primario): busca libros, realiza préstamos, los renueva y hace reservas.
  • Bibliotecario (actor primario): registra devoluciones, gestiona el inventario y administra multas.
  • Administrador (actor primario): configura el sistema y genera reportes de actividad.
  • Sistema de Email (actor secundario): envía notificaciones automáticas de vencimientos y disponibilidad.

Los tres casos de uso a especificar en detalle son:

  • CU-010 — Gestionar Usuarios: Actor accede a la gestión de usuarios, busca, edita, suspende o reactiva cuentas.
  • CU-011 — Notificar Disponibilidad: El sistema detecta la devolución de un libro con reservas activas y notifica al primer usuario en cola.
  • CU-012 — Enviar Recordatorio de Devolución: Job nocturno detecta préstamos próximos a vencer y envía recordatorios automáticos.

Usa el formato de especificación de caso de uso de IEEE 29148:2018: tabla con ID, Nombre, Actor principal, Precondición(es), Postcondición(es), Flujo principal numerado y Flujos alternativos referenciados.

Especificaciones de Casos de Uso Expandidos (IEEE 29148:2018)
CU-010 — Gestionar Usuarios
IDCU-010
Actor PrincipalAdministrador, Bibliotecario
PrecondicionesAutenticado con rol Bibliotecario o Administrador
PostcondicionesUsuario modificado, suspendido o creado en el sistema

Flujo Principal:

  1. Actor accede a "Gestión de Usuarios".
  2. Sistema muestra lista paginada de usuarios con filtros.
  3. Actor busca usuario por nombre, email o carné.
  4. Sistema muestra ficha del usuario con préstamos activos y multas.
  5. Actor selecciona acción: editar datos / suspender / reactivar.
  6. Sistema aplica el cambio y registra en log de auditoría.

Flujo Alternativo — Suspender usuario (paso 5a):

  1. 5a.1 — Actor selecciona "Suspender cuenta".
  2. 5a.2 — Sistema solicita motivo de suspensión.
  3. 5a.3 — Actor confirma la acción.
  4. 5a.4 — Sistema suspende cuenta: user.active = false.
  5. 5a.5 — Sistema cancela todas las reservas activas del usuario.

Relaciones: <<extend>> Suspender Usuario (condición: actor selecciona esa acción).

CU-011 — Notificar Disponibilidad
IDCU-011
Actor PrincipalSistema (job automático activado en devolución)
TriggerEvento de devolución de libro con reservas activas
PrecondicionesExiste al menos una reserva activa para el libro devuelto
PostcondicionesUsuario #1 en la cola notificado; contador de 48 h iniciado

Flujo Principal:

  1. Sistema detecta evento book.returned.
  2. Sistema consulta cola de reservas del libro.
  3. Sistema toma primer usuario en cola (posición = 1).
  4. Sistema envía email de notificación con link para confirmar préstamo.
  5. Sistema registra timestamp de notificación.
  6. Sistema agenda job de verificación en 48 horas.

Flujo Alternativo — Usuario no confirma en 48 h (paso 6a):

  1. 6a.1 — Job de verificación detecta que el usuario no confirmó.
  2. 6a.2 — Sistema libera el libro para el siguiente en la cola.
  3. 6a.3 — Sistema repite el proceso con el usuario en posición 2.

Relaciones: <<include>> Verificar Disponibilidad (siempre). Actor secundario: Sistema de Email.

CU-012 — Enviar Recordatorio de Devolución
IDCU-012
Actor PrincipalSistema (job nocturno, ejecuta diariamente a las 08:00)
PrecondicionesExisten préstamos con vencimiento en 3 días o en 1 día
PostcondicionesEmail de recordatorio enviado a cada usuario afectado; log de notificaciones actualizado

Flujo Principal:

  1. Job nocturno se ejecuta a las 08:00.
  2. Sistema consulta préstamos con dueDate = today + 3 días.
  3. Sistema envía email de recordatorio a cada usuario con detalles del préstamo y opción de renovación.
  4. Sistema repite para préstamos con dueDate = today + 1 día (recordatorio urgente).
  5. Sistema registra cada envío en log de notificaciones.

Flujo Alternativo — Sin préstamos próximos a vencer (paso 2a):

  1. 2a.1 — La consulta devuelve lista vacía.
  2. 2a.2 — Sistema registra en log: "Sin recordatorios pendientes para hoy".
  3. 2a.3 — Job finaliza sin enviar emails.

Flujo Alternativo — Fallo al enviar email (paso 3b):

  1. 3b.1 — El servicio de email devuelve error.
  2. 3b.2 — Sistema registra el error y programa reintento en 1 hora.
  3. 3b.3 — Sistema continúa con el siguiente usuario de la lista.

Relaciones: <<include>> Detectar Vencimientos (siempre). Actor secundario: Sistema de Email.

🎓 Conceptos Clave
Concepto Descripción breve
Caso de usoSecuencia de interacciones entre un actor y el sistema para lograr un objetivo de negocio observable
Caso de uso expandidoEspecificación detallada con flujo principal, flujos alternativos, pre/postcondiciones y relaciones
Actor primarioInicia la interacción con el sistema para lograr su objetivo (ej. Lector, Bibliotecario)
Actor secundarioApoya al sistema para cumplir el caso de uso; no inicia la interacción (ej. Sistema de Email)
PrecondiciónEstado que debe ser verdadero antes de que el caso de uso comience a ejecutarse
PostcondiciónEstado garantizado al finalizar exitosamente el caso de uso
Flujo principalSecuencia de pasos numerados del escenario más común (camino feliz)
Flujo alternativoVariación del flujo principal que ocurre cuando una condición específica se cumple o algo falla
<<include>>Relación que indica que un caso de uso siempre incorpora el comportamiento de otro (obligatorio)
<<extend>>Relación que indica que un caso de uso puede incorporar comportamiento adicional bajo una condición específica (opcional)
<<generalize>>Relación de herencia entre actores o entre casos de uso; el hijo hereda el comportamiento del padre
IEEE 29148:2018Estándar internacional de ingeniería de requisitos que define el formato y contenido mínimo de las especificaciones de casos de uso
Trazabilidad de requisitosCapacidad de vincular cada caso de uso con el requisito funcional que satisface
