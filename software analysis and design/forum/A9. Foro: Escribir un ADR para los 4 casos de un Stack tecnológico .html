A9. Foro: Escribir un ADR para los 4 casos de un Stack Tecnológico

📋 Descripción

Aplicando el formato estándar de Architecture Decision Records (ADR), redacta una decisión arquitectónica para uno de los 4 escenarios del Sistema de Biblioteca Digital. Tu ADR debe documentar el contexto técnico, la decisión tomada, las consecuencias positivas y negativas, y al menos dos alternativas evaluadas con criterios concretos. Luego comparte y defiende tu ADR en el foro.

🎯 Objetivos de Aprendizaje
  1. Documentar decisiones de diseño arquitectónico mediante el formato ADR estándar.
  2. Justificar la selección de un stack tecnológico considerando contexto, escala y restricciones del proyecto.
  3. Evaluar alternativas de arquitectura con criterios técnicos explícitos (trade-off analysis).
  4. Argumentar y defender decisiones arquitectónicas ante criterios técnicos distintos propuestos por compañeros.
📝 Requerimientos del Foro

Tu publicación en el foro debe incluir un ADR completo con los siguientes campos:

Campo ADR Descripción requerida
Título + Estado Nombre descriptivo de la decisión y estado (Propuesto / Aceptado / Rechazado)
Contexto Descripción del problema técnico y el escenario (A, B, C o D) que motiva la decisión
Decisión Opción seleccionada con justificación técnica específica al contexto
Consecuencias Al menos 2 consecuencias positivas y 2 negativas concretas y verificables
Alternativas consideradas Mínimo 2 alternativas evaluadas con criterio técnico que explique por qué fueron descartadas
Participación Comentar el ADR de al menos un compañero: señala fortalezas, debilidades o propone una alternativa no considerada
Escenarios del caso
Escenario Descripción
A — Startup MVP Equipo de 3 desarrolladores, primer lanzamiento, ~1 000 usuarios esperados, presupuesto reducido.
B — Sistema bancario crítico Alta disponibilidad 99.9%, equipos independientes por módulo, 50 desarrolladores, regulaciones de auditoría (PCI-DSS).
C — Sistema interno RRHH Empresa mediana, 200 empleados, bajo presupuesto, cambios de negocio frecuentes, integraciones con sistemas legados.
D — E-commerce estacional Carga variable (x10–50 en temporadas altas), bajo costo en reposo, escalado automático requerido, checkout y pagos críticos.

💡 Solución de Referencia — 4 ADRs del Sistema de Biblioteca Digital
ADR-001: Selección de Base de Datos — PostgreSQL vs MongoDB

Estado: Aceptado

Contexto: El Sistema de Biblioteca Digital necesita persistir datos con relaciones bien definidas (libros → préstamos → usuarios → multas). Los reportes requieren consultas con múltiples JOINs y la integridad referencial es crítica para evitar préstamos duplicados en condiciones de concurrencia.

Decisión: Se utilizará PostgreSQL como base de datos principal.

Consecuencias positivas:

  • Integridad referencial garantizada por foreign keys (crítico para préstamos y multas).
  • Soporte nativo para transacciones ACID; evita préstamos duplicados en condiciones de carrera.
  • Consultas complejas con JOINs para reportes de ocupación y estadísticas.
  • Ecosistema maduro con Prisma ORM para TypeScript.

Consecuencias negativas:

  • Esquema rígido: cambios en la estructura requieren migraciones.
  • Menor flexibilidad para almacenar metadatos variables por tipo de libro.
  • Escalado horizontal más complejo que NoSQL (sharding requiere configuración adicional).

Alternativas consideradas:

Alternativa Razón de descarte
MongoDB Las relaciones entre entidades son complejas; el modelo de documentos generaría datos duplicados (usuario embebido en cada préstamo). Las transacciones multi-documento son menos eficientes para este caso.
MySQL Funcionalidades similares, pero PostgreSQL tiene mejor soporte para tipos de datos avanzados (JSON, arrays) útiles para metadatos de libros.
ADR-002: Estrategia de Autenticación — JWT vs Sessions

Estado: Aceptado

Contexto: El sistema necesita autenticar usuarios con tres roles (lector, bibliotecario, administrador), soportar API REST consumida por web y futura app móvil, y permitir invalidación de sesiones ante compromiso de cuenta. Los tiempos de expiración deben ser distintos por rol (lectores: 7 días; admin: 8 horas).

Decisión: Se utilizará JWT con estrategia de Access Token + Refresh Token. Access Token de 15 min (rol en payload); Refresh Token de 7 días/8 h guardado en HttpOnly cookie y BD para revocación.

Consecuencias positivas:

  • Stateless: el servidor no consulta la BD en cada request (solo valida firma del JWT).
  • Multi-plataforma: funciona igual para web, móvil y futuros clientes API.
  • El rol viaja en el token; evita consultas adicionales por request.
  • Refresh tokens en BD permiten cerrar sesiones remotamente (revocación controlada).

Consecuencias negativas:

  • Gestión de dos tokens más compleja que sesiones simples.
  • Access tokens no son revocables al instante; si se comprometen, son válidos hasta expiración (15 min).
  • JWT más grande que un ID de sesión aumenta ligeramente el tamaño de cada request.

Alternativas consideradas:

Alternativa Razón de descarte
Sessions (express-session + Redis) Requiere Redis como infraestructura adicional; no es stateless (cada request hace lookup en Redis); problemas con múltiples instancias del servidor.
OAuth 2.0 externo (Google, Auth0) La biblioteca puede requerir autenticación con carné institucional, incompatible con OAuth social. Dependencia de servicio externo para función crítica.
ADR-003: Almacenamiento de Archivos — Local vs AWS S3

Estado: Aceptado

Contexto: El sistema genera reportes en PDF (semanales y mensuales) consultados ocasionalmente por el administrador. En el futuro podrían añadirse portadas de libros. El proyecto está en fase MVP con presupuesto limitado. Se aplicará la interfaz IFileStorageService (DIP) para permitir migración futura.

Decisión: Se utilizará almacenamiento local en el servidor para el MVP, protegido por la interfaz IFileStorageService que permite migrar a S3 sin cambios en el dominio.

Consecuencias positivas:

  • Costo cero en la fase inicial; no hay costos de servicios cloud.
  • Simplicidad operacional: sin credenciales ni configuración de cloud.
  • La interfaz IFileStorageService protege al dominio del cambio futuro (Inversión de Dependencias).
  • Adecuado para el volumen esperado (decenas de PDFs/mes).

Consecuencias negativas:

  • No escalable horizontalmente: si el servidor se replica, los archivos no son accesibles desde todas las instancias.
  • Sin CDN: los reportes no se sirven desde ubicaciones geográficamente cercanas.
  • Riesgo de pérdida si el servidor falla sin backup configurado.
  • Requiere migración a S3 cuando el sistema escale o se containerice.

Alternativas consideradas:

Alternativa Razón de descarte
AWS S3 Alta disponibilidad y durabilidad, pero costo operacional desde el inicio (~$0.023/GB/mes) y complejidad de configuración (IAM, bucket policies, CORS). Excesivo para el MVP.
Cloudinary Plan gratuito limitado; optimizado para imágenes. No aplica para PDFs de reportes administrativos.
ADR-004: Framework Frontend — Next.js vs Remix

Estado: Aceptado

Contexto: La interfaz web del sistema necesita páginas públicas indexables por buscadores (catálogo, búsqueda), un panel de usuario autenticado (préstamos, reservas) y un panel de administración (reportes, gestión de inventario). Debe integrarse con la API REST del backend.

Decisión: Se utilizará Next.js 14+ con App Router.

Consecuencias positivas:

  • SSR/SSG nativo: páginas de catálogo renderizadas en servidor = mejor SEO y carga inicial rápida.
  • API Routes integradas: frontend y backend pueden coexistir en un solo repositorio (fullstack).
  • Ecosistema masivo: mayor cantidad de librerías compatibles y recursos de aprendizaje en español.
  • React Server Components reducen JavaScript enviado al cliente en páginas de solo lectura.

Consecuencias negativas:

  • Complejidad del App Router: la distinción entre Server Components y Client Components tiene curva de aprendizaje.
  • Optimizado para Vercel; en otros proveedores requiere configuración adicional.
  • Bundle size mayor que Remix en aplicaciones con mucha interactividad.

Alternativas consideradas:

Alternativa Razón de descarte
Remix Excelente manejo de formularios y mutaciones, pero menor ecosistema y comunidad que Next.js; menos recursos de aprendizaje disponibles en español para el equipo.
SPA con React + Vite Sin SSR: mala experiencia para SEO del catálogo público. Requiere backend separado siempre, añadiendo complejidad de despliegue.
🎓 Conceptos Clave
Concepto Descripción breve
ADR (Architecture Decision Record) Documento breve que registra una decisión técnica importante: contexto, opciones evaluadas, decisión tomada y sus consecuencias.
Trade-off analysis Evaluación explícita de las ventajas y desventajas de cada alternativa antes de tomar una decisión de diseño.
Atributos de calidad Características no funcionales (escalabilidad, mantenibilidad, disponibilidad) que guían la selección de una arquitectura.
Arquitectura monolítica Estructura en un único despliegue (N-Tier); adecuada para equipos pequeños, MVPs y sistemas de baja complejidad.
Microservicios Sistema compuesto de servicios independientes, desplegables y escalables por separado; adecuado para equipos grandes y sistemas de alta criticidad.
Arquitectura hexagonal Patrón Ports & Adapters que aísla el núcleo de dominio de los detalles externos mediante interfaces (puertos) e implementaciones (adaptadores).
Principio DIP Inversión de Dependencias: las capas de alto nivel no dependen de capas de bajo nivel; ambas dependen de abstracciones (interfaces).
JWT (JSON Web Token) Estándar de autenticación stateless que codifica información del usuario en un token firmado; permite multi-plataforma y evita consultas de sesión por request.
Stack tecnológico Conjunto de tecnologías (lenguaje, framework, base de datos, infraestructura) que se seleccionan en función de los atributos de calidad requeridos por el sistema.
