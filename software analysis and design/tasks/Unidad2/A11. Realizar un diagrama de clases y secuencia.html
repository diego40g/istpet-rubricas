A11. Realizar un diagrama de clases y secuencia

📋 Descripción

En esta actividad modelarás el Sistema de Biblioteca Digital utilizando dos tipos de diagramas UML complementarios: el diagrama de clases, que captura la estructura estática del dominio (entidades, atributos, métodos y relaciones), y los diagramas de secuencia, que describen cómo colaboran los objetos a lo largo del tiempo para cumplir con los flujos de negocio más importantes.

🎯 Objetivos de Aprendizaje
  1. Construir un diagrama de clases UML completo con relaciones de cardinalidad, métodos con lógica de dominio y dependencias hacia interfaces.
  2. Modelar flujos de interacción mediante diagramas de secuencia UML incluyendo fragmentos combinados (alt, loop).
📝 Enunciado del caso

El dominio del Sistema de Biblioteca Digital incluye las entidades principales: Book, User, Loan, Reservation, Fine y Category. Los tres flujos de interacción a modelar son:

  • Flujo 1 — Renovar préstamo: El lector solicita renovación; el sistema verifica reservas activas y si el préstamo es renovable; actualiza o devuelve error.
  • Flujo 2 — Reservar libro: El lector reserva un ejemplar prestado; el sistema crea la reserva y notifica cuando quede disponible.
  • Flujo 3 — Generar reporte: El administrador solicita reporte de actividad; el sistema recopila estadísticas, genera el documento y lo envía.

Métodos clave con lógica de dominio 

// Book.markAsBorrowed() — verifica disponibilidad antes de prestar
markAsBorrowed(): void {
  if (this.status !== BookStatus.AVAILABLE) {
    throw new DomainError(`Book "${this.title}" is not available`);
  }
  this.status = BookStatus.BORROWED;
}
// Loan.isOverdue() — verifica si el préstamo está vencido
isOverdue(): boolean {
  if (this.status === LoanStatus.RETURNED) return false;
  return new Date() > this.dueDate;
}

// Loan.calculateFine() — calcula multa ($0.50 por día de retraso)
calculateFine(): number {
  if (!this.isOverdue()) return 0;
  const today = this.returnDate ?? new Date();
  const daysLate = Math.ceil(
    (today.getTime() - this.dueDate.getTime()) / (1000 * 60 * 60 * 24)
  );
  return daysLate * DAILY_FINE_RATE; // $0.50/día
}

// Loan.canRenew() — máximo 2 renovaciones, no vencido
canRenew(): boolean {
  return (
    this.status === LoanStatus.ACTIVE &&
    !this.isOverdue() &&
    this.renewalCount < MAX_RENEWALS // MAX_RENEWALS = 2
  );
}

// User.canBorrow() — máx 3 simultáneos, sin multas pendientes
canBorrow(): boolean {
  const activeLoans = this.getActiveLoansCount();
  const hasPendingFines = this.fines.some(f => !f.paid);
  return (
    this.active &&
    activeLoans < MAX_SIMULTANEOUS_LOANS && // MAX = 3
    !hasPendingFines
  );
}

Participantes: User · API Controller · LoanService · LoanRepository · BookRepository · ReservationRepository · NotificationService 

sequenceDiagram
    actor User
    participant API as API Controller
    participant Service as LoanService
    participant LoanRepo as LoanRepository
    participant ReservRepo as ReservationRepository

    User->>API: PATCH /loans/{loanId}/renew
    API->>Service: renewLoan(loanId, userId)
    Service->>LoanRepo: findById(loanId)
    LoanRepo-->>Service: loan

    alt loan no existe o no pertenece al usuario
        Service-->>API: throw NotFoundError
        API-->>User: 404 Not Found
    else loan existe
        Service->>ReservRepo: findByBook(loan.bookId)
        ReservRepo-->>Service: reservations[]

        alt loan.isOverdue()
            API-->>User: 400 "No se puede renovar un préstamo vencido"
        else loan.renewalCount >= MAX_RENEWALS (2)
            API-->>User: 400 "Límite de renovaciones alcanzado (máx. 2)"
        else reservations.length > 0
            API-->>User: 400 "Otro usuario está esperando este libro"
        else renovación permitida
            Service->>Service: loan.renew() [dueDate += 14 días, renewalCount++]
            Service->>LoanRepo: update(loanId, loan)
            Service->>Email: sendRenewalConfirmation(user, updatedLoan)
            API-->>User: 200 OK {loan con nueva dueDate}
        end
    end

🎓 Conceptos Clave
Concepto Descripción breve
Diagrama de Clases UML Representación estática del dominio: entidades, interfaces, enumeraciones y sus relaciones
Cardinalidad Multiplicidad de una relación (1, 0..1, 1..*, *) que indica cuántas instancias participan
Encapsulamiento Principio que oculta el estado interno de un objeto y expone solo operaciones a través de métodos públicos
Lógica de dominio Reglas de negocio implementadas como métodos dentro de las entidades (ej. Loan.isOverdue())
Interfaz de repositorio Abstracción que define las operaciones de persistencia sin depender de una tecnología de base de datos concreta
Lifeline Línea de vida en un diagrama de secuencia que representa un participante (objeto o actor) a lo largo del tiempo
Mensaje síncrono Llamada que bloquea al emisor hasta recibir respuesta (flecha sólida con punta cerrada)
Mensaje asíncrono Llamada que no bloquea al emisor; se usa para notificaciones y eventos (flecha con punta abierta)
Fragmento combinado alt Bloque de secuencia con condiciones mutuamente excluyentes, equivalente a un if/else
Fragmento combinado loop Bloque de secuencia que se repite mientras se cumpla una condición, equivalente a un while
